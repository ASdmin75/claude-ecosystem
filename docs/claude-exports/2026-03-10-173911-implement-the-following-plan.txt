вФВ                                                                              вФВ
вФВ Wizard: Auto-Generation of Tasks, Agents, Pipelines, Domains                 вФВ
вФВ                                                                              вФВ
вФВ Context                                                                      вФВ
вФВ                                                                              вФВ
вФВ Currently, creating a new automation in Claude Ecosystem requires manually   вФВ
вФВ configuring multiple entities: tasks in tasks.yaml, sub-agent .md files,     вФВ
вФВ domain schemas, pipelines вАФ and understanding how they all wire together.    вФВ
вФВ The wizard automates this: user describes what they want in natural          вФВ
вФВ language, Claude analyzes it, and generates all necessary configuration      вФВ
вФВ entities for optimal execution.                                              вФВ
вФВ                                                                              вФВ
вФВ Architecture                                                                 вФВ
вФВ                                                                              вФВ
вФВ Two-phase flow: Generate (Claude produces a structured plan via json_schema) вФВ
вФВ  вЖТ Preview/Edit вЖТ Apply (create all entities in dependency order).           вФВ
вФВ                                                                              вФВ
вФВ The wizard doesn't need its own data model вАФ it reuses existing config       вФВ
вФВ structs. Plans are held in memory with TTL (no persistence needed for a      вФВ
вФВ generateвЖТpreviewвЖТapply cycle).                                               вФВ
вФВ                                                                              вФВ

вФВ generator.go вАФ Generator struct:                                             вФВ
вФВ - NewGenerator(runner *task.Runner) *Generator                               вФВ
вФВ - Generate(ctx, req, cfg) (*Plan, error) вАФ creates a synthetic config.Task   вФВ

вФВ  output into Plan, assigns UUID + status "draft"                             вФВ
вФВ - Uses model sonnet for speed, permission_mode: plan (read-only), timeout    вФВ

вФВ                                                                              вФВ
вФВ applier.go вАФ Applier struct:                                                 вФВ
вФВ - NewApplier(cfg, subagentMgr, domainMgr) *Applier                           вФВ
вФВ - Apply(plan *Plan) (*ApplyResult, error) вАФ creates entities in strict       вФВ
вФВ order:                                                                       вФВ
вФВ   a. Domains вЖТ add to cfg.Domains, call domainMgr.AddDomain() (new method)   вФВ
вФВ   b. Agents вЖТ subagentMgr.Create() for each                                  вФВ
вФВ   c. Tasks вЖТ append to cfg.Tasks                                             вФВ
вФВ   d. Pipelines вЖТ append to cfg.Pipelines                                     вФВ
вФВ   e. Save вЖТ cfg.Save()                                                       вФВ
вФВ - Accumulates rollback functions; on any failure, executes rollbacks in      вФВ
вФВ reverse order                                                                вФВ
вФВ - Validates before mutating: no duplicate names, MCP servers exist,          вФВ
вФВ agent/task references valid                                                  вФВ
вФВ                                                                              вФВ
