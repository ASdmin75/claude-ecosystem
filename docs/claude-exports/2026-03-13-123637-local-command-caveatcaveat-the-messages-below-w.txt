❯ Давай еще немного изменим подход. Теперь будем "накпливать" потенциальных     
лидов, например в таблице БД. И когда их количество превысит какой-то порог,    
например >= 15, будем формировать Excel файл и отсылать его в ТГ и на e-mail.   
После отправки "накопленные" лиды помечаются чтобы не попасть в следующий чанк. 
 Как идея? Критикуй.                                                            

● Идея хорошая, это классический паттерн "batch accumulate & flush". Вот
  критика:

  Плюсы:
  - Не спамишь получателей одиночными лидами — один файл с 15+ записями гораздо
  полезнее
  - Excel с пачкой лидов удобнее для менеджера, чем поток отдельных сообщений
  - Контроль через БД — можно смотреть историю, статистику, повторно отправить

  Что стоит продумать:

  1. Порог 15 — это жёсткий минимум или есть таймаут? Если лиды идут медленно,
  можно ждать неделями. Нужен второй триггер: ">=15 ИЛИ прошло N дней с
  последнего флаша и есть хотя бы 1 лид".
  2. Кто триггерит проверку порога? Варианты:
    - Cron-задача (например каждые 6 часов проверяет SELECT COUNT(*) FROM leads 
  WHERE sent = 0)
    - Сразу после добавления нового лида (проще, но связывает сбор и отправку)
    - Рекомендую cron — развязывает процессы
  3. Race condition при отправке. Между SELECT лидов и UPDATE sent=1 может
  прийти новый лид. Решение: использовать транзакцию или помечать конкретные ID,
   а не "все где sent=0".
  4. Что если отправка упала? Excel сгенерился, но Telegram/email не отправился.
   Нужен статус не бинарный (sent), а трёхфазный: pending → sending → sent. Если
   застрял в sending — retry.
  5. Дедупликация лидов в таблице. Сейчас у тебя дедупликация через
  rejected_companies. А что если один и тот же лид попадёт в accumulator дважды?
   Нужен UNIQUE constraint (по домену/названию компании).
  6. Структура таблицы. Минимально:
  CREATE TABLE leads (
    id INTEGER PRIMARY KEY,
    company_name TEXT NOT NULL,
    domain TEXT UNIQUE,
    country TEXT,
    data JSON,          -- все поля лида
    status TEXT DEFAULT 'pending',  -- pending/sending/sent
    batch_id INTEGER,   -- к какой отправке относится
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    sent_at DATETIME
  );
  7. Два пайплайна вместо одного:
    - Collector (cron, частый) — парсит источники, складывает в leads
    - Flusher (cron, реже) — проверяет порог, формирует Excel, отправляет,
  помечает

  Это чище, чем всё в одном процессе.

  Итого: идея рабочая, главное добавить таймаут-триггер помимо порога и
  разделить сбор/отправку на два независимых процесса. Хочешь — могу посмотреть
  текущий код mcp-exportby и предложить конкретную реализацию?

● Понял. Тогда уточню модель:

  Ключевое: это не постоянный поток, а ручной batch-процесс. Менеджер запускает
  пайплайн, когда считает нужным (или после пополнения raw_companies). Cron —
  опционально, на потом.

  Пайплайн (единый, ручной запуск):

  1. Enrich — берёт необработанные raw_companies, Claude анализирует/фильтрует,
  годные → leads (status=pending), негодные → rejected_companies
  2. Check threshold — если pending leads >= 15 → продолжаем, иначе → стоп (или
  флаг --force для отправки любого количества)
  3. Export — генерирует Excel из pending leads
  4. Send — отправляет в ТГ + email
  5. Mark — помечает отправленные как sent
