  ---
  Проблемы

  1. find-leads: allowed_tools блокирует работу с сабагентом (КРИТИЧЕСКАЯ)

  tasks.yaml:93-94 — задача ограничивает инструменты до WebSearch и WebFetch:
  allowed_tools:
    - WebSearch
    - WebFetch
  Это передаётся как --allowedTools WebSearch WebFetch в Claude CLI. Но чтобы
  Claude мог вызвать сабагента eaeu-logistics-lead-finder (настроенного через
  --agents), ему нужен инструмент Agent. Он заблокирован — сабагент никогда не
  вызовется. Claude будет пытаться всё делать сам.

  2. find-leads: max_turns: 30 слишком мало

  25 лидов × (поиск + чтение страниц) = минимум 50-75 тёрнов. При 30 тёрнах
  Claude остановится на полпути.

  3. compile-leads-excel: {{.Date}} не передаётся в пайплайне (БАГ)

  pipelines.go:96-98 передаёт только PrevOutput:
  vars := map[string]string{
      "PrevOutput": lastOutput,
  }
  Переменная Date не заполняется → файл будет называться leads-report-.xlsx
  (пустая дата).

  4. Пайплайн не применяет per-task таймауты (ПОТЕНЦИАЛЬНЫЙ ЗАВИСОН)

  В pipelines.go:100 задача запускается с контекстом пайплайна, а не с таймаутом
   задачи:
  result := s.taskRunner.Run(ctx, *t, opts, vars)
  В отличие от handleRunTask, который делает context.WithTimeout(ctx, timeout).
  Если find-leads зависнет — весь пайплайн зависнет навсегда.

  5. MCP серверы без env в конфиге

  tasks.yaml:169-177 — MCP серверы не имеют env в YAML. .env подгружается в
  процесс сервера, и Claude CLI наследует переменные окружения от родителя. Это
  работает, но только если Claude CLI передаёт окружение своим MCP-процессам.
  Стоит явно добавить env для надёжности.
