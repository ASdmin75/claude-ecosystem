  - mode: parallel — запускает все steps одновременно через goroutines +

  1. Mixed mode (DAG) — граф зависимостей, где часть шагов параллельна, а часть
  зависит от предыдущих:
  pipelines:
    - name: review-pipeline
      mode: dag
      steps:
        - task: lint-code
          id: lint
        - task: security-scan
          id: security
        - task: review-code        # запустится только после lint И security
          id: review
          depends_on: [lint, security]
        - task: summarize
          depends_on: [review]

  2. Fan-out — один агент генерирует список, затем N параллельных агентов
  обрабатывают каждый элемент:

      splitter: list-files        # задача, возвращающая JSON-массив
      worker: review-single-file  # запускается параллельно для каждого элемента
      collector: merge-reviews
      max_parallel: 5

  3. Shared context через файл/БД — параллельные агенты пишут в общий domain,
  collector читает.

❯ А как A2A реализован в Claude Code ?                                          

● Claude Code поддерживает A2A (agent-to-agent) через механизм sub-agents — это
  файлы .claude/agents/*.md с YAML frontmatter.

  Как это работает в Claude Code

  Когда вы используете claude -p, агент может делегировать работу другим агентам
   через встроенный инструмент Agent. Claude Code сам решает, когда вызвать
  sub-agent, на основе описания в frontmatter.

  Пример файла .claude/agents/reviewer.md:
  ---
  name: reviewer
  description: Reviews code for bugs and style issues
  model: sonnet
  tools:
    - Read
    - Grep
    - Glob
  ---

  You are a code reviewer. Analyze the provided code...

  Главный агент вызывает sub-agent так:
  Tool: Agent
  {
    "name": "reviewer",
    "prompt": "Review the changes in src/api/"
  }

  Sub-agent работает в отдельном контексте, со своим набором tools, моделью и
  system prompt. Результат возвращается главному агенту как текст.

  Почему DAG и fan-out не нужны

  DAG — over-engineering для этого проекта. Реальные сценарии с claude -p:
  - Каждый вызов стоит денег и занимает минуты
  - Типичный pipeline — 2-4 шага, не граф из 15 узлов
  - Если нужен "частичный параллелизм", проще разбить на два pipeline

  Fan-out — дублирует то, что Claude уже делает сам. Если дать одному claude -p
  задачу "проревьюй 10 файлов" с sub-agents, он сам распараллелит через
  встроенный Agent tool. Запускать 10 отдельных claude -p — дороже и без общего
  контекста.

  Что реально полезно: session chaining (--resume)

  У вас уже есть ResumeSessionID в RunOptions (task/command.go:86-88), но в
  sequential pipeline он не используется. Если передавать session_id от
  предыдущего шага к следующему через --resume:

  - Агент помнит контекст предыдущего шага без дублирования в промпте
  - Не нужно передавать весь {{.PrevOutput}} (который может быть огромным)
  - Дешевле — Claude не парсит заново весь предыдущий вывод
  - Качественнее — полный контекст вместо сжатого текста

  Пример сценария:
  pipelines:
    - name: review-fix
      mode: sequential
      session_chain: true    # ← новый флаг
      steps:
        - task: code-review   # возвращает session_id
        - task: apply-fixes   # --resume с session_id предыдущего шага
