  1. Почему только 6 лидов вместо 40

  В БД сейчас 6 компаний, scan_log пуст. Причины:

  Бюджет turns исчерпывается быстро. Для каждой компании агент делает минимум
  3-4 вызова инструментов:
  1. WebFetch — каталог (1 страница)
  2. WebFetch — страница каждой компании
  3. mcp__database__query — проверка на дубль
  4. mcp__database__execute — INSERT

  Для 40 лидов с 10 страниц каталога это ~50-60 вызовов инструментов, а
  max_turns: 40 + вложенный агент (у которого тоже maxTurns: 40, но он вложен в
  основной, который имеет свой лимит). При этом основной таск делегирует агенту
  через Agent tool — это дополнительный overhead.

  Бюджет $0.5 — очень мало для Sonnet с десятками WebFetch (каждый может вернуть
   большой HTML).

  Архитектурная проблема — Claude как скрейпер крайне неэффективен. Каждый fetch
   страницы — это полный round-trip через LLM, плюс весь HTML идёт в контекст.

  2. Как организовать работу без перезапуска с начала

  Ваша идея верна — нужно разделить скрейпинг и анализ. Предлагаю такую
  архитектуру:

  Новая схема: 2-уровневый пайплайн

  Уровень 1: Сбор сырых данных (bulk scrape)

  Добавить в схему таблицу raw_companies и таблицу scan_progress:

  CREATE TABLE IF NOT EXISTS raw_companies (
      id INTEGER PRIMARY KEY AUTOINCREMENT,
      name TEXT NOT NULL UNIQUE,
      url TEXT,
      brief TEXT,            -- краткое описание с каталога
      detail_html TEXT,      -- содержимое страницы компании
      products TEXT,
      contact_info TEXT,
      export_destinations TEXT,
      scraped_at TEXT,
      enriched INTEGER DEFAULT 0,  -- 0=только каталог, 1=с деталями
      analyzed INTEGER DEFAULT 0   -- 0=не оценена, 1=оценена
  );

  CREATE TABLE IF NOT EXISTS scan_progress (
      id INTEGER PRIMARY KEY AUTOINCREMENT,
      last_page INTEGER NOT NULL,
      total_pages INTEGER,
      scanned_at TEXT NOT NULL
  );

  Уровень 2: Разделённые задачи

  Задача: scrape-export-by-catalog
  Что делает: Сканирует страницы каталога, сохраняет name+url+brief в
    raw_companies. Проверяет scan_progress и продолжает с последней страницы
  Частота: 2 раза/неделю
  ────────────────────────────────────────
  Задача: enrich-export-by-companies
  Что делает: Берёт из raw_companies WHERE enriched=0 LIMIT 20, загружает
    детальные страницы, обновляет products/contacts/destinations
  Частота: ежедневно
  ────────────────────────────────────────
  Задача: analyze-export-by-aviation
  Что делает: Берёт raw_companies WHERE analyzed=0, Claude оценивает
    aviation_priority без WebFetch — только на основе данных в БД, записывает в
    companies
  Частота: после enrich
  ────────────────────────────────────────
  Задача: check-export-by-new
  Что делает: Смотрит только page=1 каталога, ищет новые записи
  Частота: ежедневно

  Ключевые преимущества:

  1. Продолжение с последней страницы — scan_progress хранит номер последней
  просканированной страницы
  2. Анализ без WebFetch — самая дорогая часть (оценка aviation_priority)
  работает только с текстом из БД, что в 10x дешевле
  3. Инкрементальность — enriched=0 и analyzed=0 гарантируют обработку только
  новых записей
  4. Проверка обновлений — отдельная лёгкая задача смотрит только первую
  страницу
