     Как создать пайплайн для 4Logist API через Wizard

     Context

     Пользователь добавил спецификацию API 4Logist (specs/4log.json) и
     env-переменные (4LOG_CLIENT_ID, 4LOG_CLIENT_SECRET, 4LOG_API_URL) в .env.
     Нужно создать пайплайн для формирования отчётов и мониторинга заказов через
      wizard в Web UI.

     Инструкция по использованию Wizard

     1. Запустить сервер

     make build && make build-ui && make run

     Открыть Web UI (по умолчанию http://localhost:8080).

     2. Открыть Wizard

     В боковом меню нажать на кнопку Wizard (иконка волшебной палочки).

     3. Ввести описание

     В поле описания ввести примерно следующее (на русском или английском):

     ---
     Вариант описания для wizard:

     Создай пайплайн для работы с транспортной CRM 4Logist.

     API спецификация: specs/4log.json
     API URL: ${4LOG_API_URL} (из .env)
     Авторизация: OAuth2 client_credentials, token URL:
     ${4LOG_API_URL}/oauth/v2/token,
     client_id из env: ${4LOG_CLIENT_ID}, client_secret из env:
     ${4LOG_CLIENT_SECRET}.

     Нужно два пайплайна:

     1. Пайплайн "4log-daily-report" (ежедневный отчёт):
        - Шаг 1: Загрузить заказы за вчера (post_api_order_list), клиентов
     (post_api_client_list),
          счета (post_api_invoices_list) — сохранить в локальную БД (домен
     4logist)
        - Шаг 2: Сформировать Excel-отчёт с листами: заказы, клиенты, финансы
        - Шаг 3: Отправить отчёт в Telegram
        Расписание: каждый день в 9:00 (cron: 0 9 * * *)

     2. Пайплайн "4log-order-monitor" (мониторинг заказов):
        - Шаг 1: Получить активные заказы и их статусы (post_api_order_list,
     post_api_order_statuses_list)
        - Шаг 2: Сравнить с предыдущим состоянием в БД, найти изменения статусов
        - Шаг 3: При наличии изменений — отправить уведомление в Telegram
        Расписание: каждые 30 минут (cron: */30 * * * *)

     MCP серверы: openapi (для 4log API), database (SQLite), excel, telegram,
     filesystem.
     Домен: 4logist (БД для хранения заказов, клиентов, счетов, статусов).

     ---
     4. Что сгенерирует Wizard

     Wizard вызовет Claude, который создаст план со следующими сущностями:

     - MCP Server 4log-api: ./bin/mcp-openapi с env-переменными:
       - OPENAPI_SPEC_PATH=specs/4log.json
       - OPENAPI_BASE_URL=${4LOG_API_URL}
       - OPENAPI_AUTH_TYPE=oauth2_client_credentials
       - OPENAPI_OAUTH2_TOKEN_URL=${4LOG_API_URL}/oauth/v2/token
       - OPENAPI_OAUTH2_CLIENT_ID=${4LOG_CLIENT_ID}
       - OPENAPI_OAUTH2_CLIENT_SECRET=${4LOG_CLIENT_SECRET}
     - Domain 4logist: SQLite БД со схемой для заказов, клиентов, статусов
     - Tasks: 5-6 задач (fetch-orders, fetch-clients, build-report,
     monitor-orders, notify-changes и т.д.)
     - Pipelines: 2 пайплайна (daily-report, order-monitor)

     5. Preview и Apply

     На экране Preview:
     - Развернуть каждую секцию (MCP Servers, Domains, Agents, Tasks, Pipelines)
     - Проверить что OAuth2 параметры корректны
     - Важно: убедиться что OPENAPI_OAUTH2_TOKEN_URL указан правильно (это может
      быть /oauth/v2/token или другой путь — зависит от 4Logist)
     - Можно отредактировать план кнопкой Edit перед применением
     - Нажать Apply для создания всех сущностей

     6. После Apply

     - Wizard создаст всё в tasks.yaml в правильном порядке зависимостей
     - Сгенерирует SETUP.md с перечнем необходимых env-переменных
     - Scheduler автоматически подхватит cron-расписания

     Важные замечания

     - Token URL: если стандартный /oauth/v2/token не работает, нужно уточнить
     правильный URL в документации 4Logist и поправить в конфиге MCP-сервера в
     tasks.yaml
     - env-переменные: используйте ${VAR} синтаксис в конфиге MCP-сервера —
     значения подставятся из .env
     - allowed_tools: wizard сгенерирует список разрешённых tools на основе
     операций из спецификации (например mcp__4log-api__post_api_order_list)
     - Тестирование: после Apply можно запустить отдельную задачу через UI или
     make run-task TASK=fetch-4log-orders для проверки подключения к API
