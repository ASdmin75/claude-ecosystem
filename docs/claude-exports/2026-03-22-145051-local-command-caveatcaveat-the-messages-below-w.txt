‚óŹ Found the bug. In cleanDomainRefs (delete.go:114), the orphan check includes
  len(d.MCPServers) == 0, but MCP servers in domain reference lists are shared
  infrastructure (like database, excel). They're almost never exclusively owned
  by deleted tasks, so they're never cleaned from the domain's list, preventing
  the domain from being detected as orphaned.

  keep it alive. If all tasks, pipelines, and agents are gone, the domain is
  orphaned regardless of MCP servers.

  Root cause: In cleanDomainRefs, the orphan check required len(d.MCPServers) ==
   0 to delete a domain. But MCP servers in domain reference lists (like
  database, excel) are shared infrastructure used by many domains/tasks. When a
  pipeline is cascade-deleted, the exclusive MCP check filters out shared
  servers, so they're never removed from the domain's mcp_servers list ‚ÄĒ
  preventing the domain from ever being considered orphaned.

  Fix: Removed len(d.MCPServers) == 0 from the orphan condition in
  delete.go:114. MCP servers are tools the domain uses, not entities that should
   keep it alive. If all tasks, pipelines, and agents are gone, the domain is
  orphaned regardless.
