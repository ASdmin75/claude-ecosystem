  ## Code Review Summary                                                    
                                                                            
  ### Error Handling                                                        
                                                                            
  1. **`cmd/hook/main.go:50` — Ignored JSON encode error.**                 
  `json.NewEncoder(os.Stdout).Encode(output)` can fail but the error is     
  silently discarded. Same issue on the `Bash` block path (line 50) vs the  
  default approve path (line 63) which uses raw `fmt.Fprint` — inconsistent 
   encoding approaches.                                                     
                                                                            
  2. **`scheduler/scheduler.go:37` — No timeout on agent context.**         
  Scheduled agents use `context.Background()` with no deadline, meaning a   
  hung `claude` process will block the cron slot indefinitely. The watcher  
  has the same issue — it passes the parent ctx but never applies a per-run 
   timeout.                                                                 
                                                                            
  3. **`config/config.go` — No validation.** Agent names aren't checked for 
   uniqueness; `WorkDir` isn't validated to exist; `Watch.Debounce` is      
  declared as a string but never parsed or used (the watcher hardcodes 3s). 
                                                                            
  ### Concurrency Safety                                                    
                                                                            
  4. **`watcher/watcher.go:58` — `debounce` map is racy.** The `debounce`   
  map is read/written in the main select loop, which is fine, but the       
  `w.agents` slice is appended in `Register` (line 42) and iterated in      
  `Start` (line 78). If `Register` and `Start` can overlap, this is a data  
  race. No mutex protects `w.agents`.                                       
                                                                            
  5. **`events/bus.go:32-34` — Fire-and-forget goroutines.** Every handler  
  is launched as an untracked goroutine (`go h(e)`). There's no mechanism   
  to wait for handlers to finish, which means the program can exit (or      
  shutdown can complete) while handlers are still running. A                
  `sync.WaitGroup` would fix this.                                          
                                                                            
  6. **`watcher/watcher.go:85` — Unbounded goroutine spawning.** Each file  
  change event spawns a goroutine per matching agent with no concurrency    
  limit. A burst of file events (e.g., `git checkout`) could spawn hundreds 
   of concurrent `claude` subprocesses.                                     
                                                                            
  ### Idiomatic Go                                                          
                                                                            
  7. **`watcher/watcher.go` — `debounce` map grows unbounded.** Entries are 
   added but never cleaned up. Over a long-running daemon lifetime this is  
  a slow memory leak.                                                       
                                                                            
  8. **`agent/agent.go:14-18` — `time.Duration` in JSON.** `Duration`       
  marshals as an integer (nanoseconds) by default, which is opaque.         
  Consider a custom type or string field for human-readable output.         
                                                                            
  9. **`cmd/orchestrator/main.go:36-42` — Event handler writes to stdout    
  without synchronization.** The `fmt.Fprintf(os.Stdout, ...)` call runs in 
   a goroutine (via the event bus) and could interleave with `fmt.Println`  
  on line 53 or other output.                                               
                                                                            
  10. **`cmd/hook/main.go:66` — Blocklist is easy to bypass.** `rm -rf /`   
  is blocked but `rm -rf /*`, `rm -r -f /`, or `sudo rm -rf /` are not.     
  Pattern matching on raw command strings is inherently fragile.            
                                                                            
  ### Priorities                                                            
                                                                            
  The most impactful issues to fix are: **unbounded goroutine spawning      
  (#6)**, **no agent timeout (#2)**, and **fire-and-forget handlers without 
   shutdown coordination (#5)**. These directly affect production           
  reliability. The debounce map race (#4) and memory leak (#7) are          
  secondary but straightforward fixes.

  ## Code Review Summary                                                    
                                                                            
  Overall the codebase is clean and well-structured for its size. Here are  
  the findings:                                                             
                                                                            
  ### Error Handling                                                        
                                                                            
  1. **`config/config.go:37-39`** — `ParsedTimeout()` and                   
  `ParsedDebounce()` silently swallow parse errors, returning defaults. A   
  malformed duration like `timeout: "abc"` would be invisible to the user.  
  Consider logging or returning the error during config validation instead. 
                                                                            
  2. **`config/config.go:82-91`** — `validate()` checks for name uniqueness 
   but doesn't verify that each agent has a `prompt`, `work_dir`, or at     
  least one trigger (`schedule`/`watch`). An agent with no schedule and no  
  watch would be silently ignored.                                          
                                                                            
  3. **`cmd/orchestrator/main.go:81-89`** — Registration errors for         
  scheduled/watcher agents are logged but execution continues. This is      
  reasonable for a daemon, but a single bad agent config could be hard to   
  notice.                                                                   
                                                                            
  4. **`cmd/orchestrator/main.go:54`** — `runner.Run(context.Background(),  
  ...)` in single-agent mode has no timeout, so a hung `claude` process     
  would block forever. The `ParsedTimeout` is not applied here.             
                                                                            
  ### Concurrency Safety                                                    
                                                                            
  5. **`internal/events/bus.go:28-40`** — **Race between `Publish` and      
  `Wait`**: If `Publish` is called concurrently, there's a window between   
  `RUnlock` (line 31) and `wg.Add(1)` (line 34) where `Wait()` could return 
   prematurely because the WaitGroup counter is momentarily zero. This is a 
   known `sync.WaitGroup` pitfall — `Add` must be called before the

  panics. In practice this is unlikely given the usage pattern, but it's

  between `RUnlock` and the goroutine launches, but `Add` is inside the

  (`scheduler/scheduler.go:19`) — This goes against the Go convention that  
  contexts should be passed as function parameters, not stored in structs.  
  The `Scheduler` stores both `ctx` and `cancel`. A more idiomatic approach 
   would pass context through `Start(ctx)` and derive child contexts in the 
   cron jobs.                                                               
                                                                            
  8. **Error as string in `Result`** (`agent/agent.go:18`) — Using `Error
