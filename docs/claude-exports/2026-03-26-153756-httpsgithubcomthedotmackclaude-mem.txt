● Хороший вопрос. Вот мой анализ:

  claude-mem — стоит ли?

  Что это делает

  Система persistent memory для Claude Code: хуки на каждом этапе сессии
  автоматически записывают наблюдения в SQLite + Chroma (vector DB), а потом при
   поиске выдают контекст через 3-уровневую прогрессивную загрузку.

  Скорее наоборот — добавит расход, а не сэкономит

  Почему:

  1. Хуки на каждом шаге — PostToolUse, SessionEnd и т.д. каждый раз вызывают
  обработку и запись. Это overhead на каждое действие.
  2. У тебя уже есть встроенная memory-система — Claude Code имеет auto memory
  (файлы в .claude/projects/.../memory/), которую ты уже используешь. Она
  бесплатна по токенам — просто читает файлы при старте.
  3. Доп. зависимости — Bun, Python (uv), Chroma vector DB, HTTP-сервис на порту
   37777. Это целый стек ради memory.
  4. Заявленная экономия "10x" — это по сравнению с наивным подходом "загрузить
  всю историю в контекст". Но встроенная система Claude Code и так не загружает
  всю историю — она читает только MEMORY.md индекс и отдельные файлы по
  необходимости.
  5. Semantic search vs. простые файлы — vector search полезен при
  сотнях/тысячах записей. Для типичного проекта 10-30 memory-файлов с понятными
  именами — grep/glob работает не хуже.

  Когда это могло бы помочь

  - Если у тебя огромный проект с десятками разработчиков и месяцами истории
  - Если нужно автоматически захватывать контекст без ручного сохранения
  - Если встроенной memory не хватает по объёму

  Вывод

  Для твоего use case (solo-разработчик, один проект claude-ecosystem) — это 
  overkill. Встроенная memory-система Claude Code делает то же самое проще и без
   лишних токенов. claude-mem добавит complexity и зависимости, а реальной
  экономии не даст.

● Справедливый вопрос. Во встроенной системе "ручное" — это два варианта:

  1. Ты просишь явно: "запомни, что X" — и я сохраняю в memory-файл
  2. Я решаю сам в процессе разговора, что что-то стоит сохранить (роль
  пользователя, фидбек, контекст проекта) — и записываю

  В обоих случаях это осознанный выбор, а не автоматический захват всего подряд.

  Разница с claude-mem

  ┌──────────────┬──────────────────────────┬──────────────────────────────┐
  │              │    Встроенная memory     │          claude-mem          │
  ├──────────────┼──────────────────────────┼──────────────────────────────┤
  │ Что          │ Только то, что явно      │ Каждое использование         │
  │ сохраняется  │ решено сохранить         │ инструмента, каждая сессия   │
  ├──────────────┼──────────────────────────┼──────────────────────────────┤
  │ Когда        │ По ходу разговора,       │ Автоматически на хуках       │
  │              │ выборочно                │                              │
  ├──────────────┼──────────────────────────┼──────────────────────────────┤
  │ Формат       │ Markdown-файлы           │ SQLite + vector embeddings   │
  ├──────────────┼──────────────────────────┼──────────────────────────────┤
  │ Шум          │ Мало — только важное     │ Много — потом фильтруется    │
  │              │                          │ при поиске                   │
  └──────────────┴──────────────────────────┴──────────────────────────────┘

  Компромисс

  Автозахват claude-mem — это и плюс, и минус. Плюс: ничего не потеряется.
  Минус: 90% записей — шум ("прочитал файл X", "запустил тест"), который потом
  тратит токены на поиск и фильтрацию.

  Встроенная система теряет детали (если я не догадался сохранить), но зато
  memory чистая и компактная — каждая запись несёт смысл.

  Так что ты прав — "вручную" здесь означает "не автоматически", а не то что
  нужно что-то делать руками. Просто если считаешь, что я что-то важное упускаю
  — скажи "запомни это", и оно сохранится.
