● Анализ: Memory vs отдельное хранилище для лидов

  Как сейчас работает memory

  Сабагент eaeu-logistics-lead-finder имеет доступ к memory между запусками:

  - .claude/agent-memory/eaeu-logistics-lead-finder/MEMORY.md — автоматически
  загружается в контекст (первые 200 строк)
  - Детальные файлы (leads-belarus-pharma-p1.md, etc.) — ~3300 строк, 100
  компаний с полными профилями
  - Агент может читать эти файлы через Grep/Read и должен проверять их перед
  поиском новых лидов

  Проблема: memory недостаточно для дедупликации

  Есть несколько существенных ограничений:

  1. MEMORY.md ограничен 200 строками — туда не влезет даже список названий 500+
   компаний
  2. Детальные файлы доступны, но не гарантированно читаются — агент может их
  прочитать, но это зависит от его "дисциплины" и количества max_turns
  3. Нет автоматической проверки — в коде orchestrator'а нет логики
  дедупликации, всё на совести LLM
  4. Контекстное окно — при 100+ компаниях файлы лидов занимают 3300+ строк,
  загрузка всех перед поиском съедает значительную часть контекста
  5. Ненадёжность LLM — даже прочитав список, модель может пропустить дубликат
  (особенно если название компании немного отличается)

  Рекомендация

  Для надёжной дедупликации стоит создать отдельное хранилище — и у нас уже есть
   SQLite (internal/store/sqlite/). Варианты:

● RAG здесь — overkill. Вот почему:

  Что делает RAG

  RAG нужен когда у вас неструктурированные данные (документы, статьи,
  переписки) и вы ищете семантически похожее по смыслу. Это требует:
  - Vector DB (Qdrant, Chroma, pgvector)
  - Embedding модель
  - Инфраструктуру для индексации

  Почему для лидов это не нужно

  Лиды — это структурированные данные с чёткими полями:
  - Название компании
  - ИНН/УНП
  - Страна, отрасль
  - Контакты

  Дедупликация здесь — это точное или fuzzy совпадение по ключевым полям, а не
  семантический поиск. Вопрос "встречалась ли эта компания раньше?" решается
  простым SELECT ... WHERE name LIKE '%..%' OR tax_id = '...'.

  Сравнение

  ┌─────────────────────┬────────────────────┬──────────────────────────────┐
  │      Критерий       │       SQLite       │             RAG              │
  ├─────────────────────┼────────────────────┼──────────────────────────────┤
  │ Сложность           │ Таблица + 2-3      │ Vector DB + embeddings +     │
  │                     │ запроса            │ pipeline                     │
  ├─────────────────────┼────────────────────┼──────────────────────────────┤
  │ Точность            │ Высокая (точные    │ Избыточная для структурных   │
  │ дедупликации        │ поля)              │ данных                       │
  ├─────────────────────┼────────────────────┼──────────────────────────────┤
  │ Инфраструктура      │ Уже есть в проекте │ Новая зависимость            │
  ├─────────────────────┼────────────────────┼──────────────────────────────┤
  │ Скорость            │ Мгновенно          │ Задержка на embedding        │
  ├─────────────────────┼────────────────────┼──────────────────────────────┤
  │ Поддержка           │ Минимальная        │ Значительная                 │
  └─────────────────────┴────────────────────┴──────────────────────────────┘

  Когда RAG был бы уместен

  Если бы задача была другой — например, искать лидов по похожести на

● Отличная идея. Разберём.

  Domain MD файл — что это даёт

  data/leads/
  ├── leads.db
  ├── DOMAIN.md          ← описание домена для AI

  DOMAIN.md содержит:
  - Описание домена и его назначение
  - Схему таблиц (какие поля, что означают)
  - Доступные MCP-инструменты и как ими пользоваться
  - Правила дедупликации (по каким полям проверять)
  - Примеры вызовов инструментов

  Это упростит работу AI

  Почему: Сейчас агент должен "знать" как работать с доменом из своего промпта в
   .claude/agents/*.md. Если добавить новый домен, нужно переписывать промпт
  агента. С DOMAIN.md:

  1. Самодокументируемость — агент читает DOMAIN.md и сразу знает что за
  таблицы, как проверять дубли, какие инструменты вызывать
  2. Переиспользуемость — разные агенты работают с одним доменом без
  дублирования инструкций в каждом промпте
  3. Автоматический контекст — система может инжектировать содержимое DOMAIN.md
  в append_system_prompt задачи, агенту не нужно его искать

  Как это подключить

  В tasks.yaml домен может указывать:
  domains:
    leads:
      data_dir: data/leads
      db: leads.db
      domain_doc: DOMAIN.md    # ← относительно data_dir
      schema: |
        CREATE TABLE IF NOT EXISTS leads (...)

  При запуске задачи с domain: leads система:
  1. Читает data/leads/DOMAIN.md
  2. Добавляет его содержимое в append_system_prompt задачи (или передаёт через
  env DOMAIN_DOC_PATH)

  Агенту не нужно ничего знать заранее — он получает документацию домена
  автоматически.

  Возможный минус

  Дополнительный расход контекстного окна на DOMAIN.md. Но если документ
  компактный (50-100 строк), это незначительно и окупается точностью работы
  агента.
