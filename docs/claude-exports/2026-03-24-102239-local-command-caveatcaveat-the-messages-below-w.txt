     Context

     Комплексный аудит проекта выявил ~40 проблем: race conditions, утечки
     памяти, уязвимости безопасности, отсутствие валидации, пробелы в
     тестировании. Цель — исправить все найденные проблемы и покрыть исправления
      тестами.

     ---
     Фаза 1: CRITICAL — Race Conditions & Resource Leaks

     1.1 Race condition в hot-reload конфига

     - Файл: cmd/server/main.go (строки 340-389)
     - Проблема: cfg.Tasks и cfg.Pipelines мутируются без синхронизации;
     scheduler/watcher/API читают их конкурентно
     - Фикс: Добавить sync.RWMutex в config.Config, обернуть все чтения/записи
     - Файлы для изменения: internal/config/config.go, cmd/server/main.go,
     internal/api/router.go, internal/scheduler/scheduler.go,
     internal/watcher/watcher.go
     - Тест: internal/config/config_test.go — concurrent read/write test с -race

     1.2 Утечка event handlers в SSE

     - Файл: internal/events/bus.go, internal/api/sse.go
     - Проблема: Subscribe() без Unsubscribe() — handlers накапливаются при
     каждом подключении клиента
     - Фикс: Добавить Unsubscribe() в Bus, вернуть cleanup-функцию из Subscribe,
      вызывать при disconnect
     - Тест: internal/events/bus_test.go — subscribe/unsubscribe/leak test

     1.3 Transaction rollback в mcp-database

     - Файл: cmd/mcp/mcp-database/main.go (строки 291-338)
     - Проблема: tx.Rollback() не вызывается при ошибках
     - Фикс: defer tx.Rollback() сразу после db.Begin()
     - Тест: cmd/mcp/mcp-database/main_test.go — test batch insert с ошибками

     1.4 Утечка подписок в pipeline

     - Файл: internal/pipeline/parallel.go (строка 93)
     - Проблема: Subscribe без Unsubscribe при каждом запуске pipeline
     - Фикс: Использовать новый Unsubscribe из 1.2
     - Тест: вместе с тестами pipeline (Фаза 4)

● По умолчанию ничего не нужно менять — без настройки всё работает как раньше
  (все пути разрешены).

  Как это работает

  Новая переменная окружения ALLOWED_DIRS ограничивает, к каким директориям
  filesystem MCP может получить доступ. Если она не задана — ограничений нет.

  Настройка

  В tasks.yaml в секции mcp_servers добавьте ALLOWED_DIRS в env:

  mcp_servers:
    - name: filesystem
      command: bin/mcp-filesystem
      env:
        ALLOWED_DIRS: "/home/asdmin/development:/home/asdmin/data"

  Формат: пути через двоеточие (как PATH).

  В этом примере MCP сервер сможет читать/писать файлы только внутри:
  - /home/asdmin/development/...
  - /home/asdmin/data/...

  Попытка обратиться к /etc/passwd или /home/asdmin/../etc/shadow вернёт ошибку
  access denied: path ... is outside allowed directories.

  Когда стоит настроить

  - Если задачи работают с конкретным work_dir — укажите его
  - Если используются домены с data_dir — добавьте и его
  - Для максимальной безопасности укажите минимальный набор директорий

  Когда можно не настраивать

  - Если сервер работает только локально и задачи уже ограничены промптами
  - Если Claude CLI и так запускается в sandbox-окружении
